## 2. 项目背景

个人开发者通常同时维护多个代码项目，但缺少一个统一的视角来回答以下问题：

- 最近主要时间投入到了哪些项目。
- 哪些项目处于持续活跃状态。
- 哪些项目带来了更多关注和反馈。
- 当前主要使用的技术栈是什么，是否发生了变化。
- 接下来最值得继续经营的项目是哪一个。

GitHub 提供了仓库、提交、流量、语言和文件内容等信息，但数据分散、颗粒度不统一，不能直接形成一个适合个人使用的可视化经营盘。因此需要构建一个围绕个人开发者工作流的数据产品，将代码活动、项目活跃、技术栈画像和经营结果集中展示。

## 3. 项目目标

### 3.1 核心目标

构建一个面向个人开发者的可视化大屏，用于展示 GitHub 项目的活跃度、提交趋势、技术栈画像和经营效果，并输出简单的行动建议。

### 3.2 业务目标

- 形成一个可持续更新的个人项目总览面板。
- 帮助用户快速识别高活跃项目和低维护项目。
- 帮助用户量化个人近阶段的技术栈分布。
- 帮助用户判断哪些项目值得继续投入。
- 作为个人作品集中的代表性项目进行展示。

### 3.3 成功标准

- 用户可以查看所有纳入同步范围的 GitHub 仓库。
- 用户可以查看每个项目按天、周、月的提交趋势。
- 用户可以查看个人提交热力图。
- 用户可以查看每个项目的技术栈标签。
- 用户可以查看项目活跃排行和基础经营指标。
- 系统可以生成 3 至 5 条自动洞察。

## 4. 用户与使用场景

### 4.1 目标用户

- 主要用户：单个 GitHub 账号的个人开发者
- 次要用户：用于作品展示的招聘方、面试官、合作方

### 4.2 核心场景

- 用户打开首页，快速了解最近 30 天的整体开发状态。
- 用户查看哪些项目最近最活跃。

- 用户查看自己的技术栈结构和近期变化。
- 用户根据系统洞察判断下一步重点投入方向。

## 7. 技术方案

### 7.1 技术栈

- 前端：React + Vite + TypeScript
- 可视化：ECharts
- 后端：Node.js + TypeScript
- 服务框架：Express 或 Fastify
- 数据库：SQLite
- 定时任务：node-cron
- 数据校验：zod

### 7.2 架构说明

- Node 服务负责数据同步、数据清洗、聚合计算和 API 提供
- SQLite 负责存储原始数据和聚合结果
- React 前端负责展示和交互
- ECharts 负责图表渲染

### 7.3 数据流

1. GitHub API 返回仓库、提交、流量、语言和文件数据
2. Node 服务进行清洗、聚合和技术栈规则识别
3. 清洗后数据写入 SQLite
4. 前端通过 API 读取聚合结果
5. 页面渲染总览、趋势、热力图、排行和洞察

### 9.4 活跃度分析模块

#### 功能说明

基于提交数据，计算每个项目的按天、按周、按月活跃度指标。

#### 功能点

- 计算每个项目日提交次数
- 计算每个项目周提交次数
- 计算每个项目月提交次数
- 统计近 7 天、30 天、90 天趋势
- 统计近 30 天活跃天数
- 统计最近一次提交时间
- 计算项目活跃排行

#### 核心指标

- repo_commit_count_7d
- repo_commit_count_30d
- repo_commit_count_90d
- repo_active_days_30d
- repo_last_commit_at
- repo_trend_day
- repo_trend_week
- repo_trend_month

#### 统计口径

- 所有时间统计基于系统配置时区，默认值为 `Asia/Shanghai`
- 日统计按自然日计算
- 周统计按自然周计算，周一为每周起始日
- 月统计按自然月计算
- 活跃天数定义为过滤后提交次数大于 0 的自然日数量
- 默认不计入 merge commit
- 默认不计入 bot commit
- 首页趋势图、项目趋势图和热力图统一使用同一套过滤口径，避免跨页面数据不一致

### 9.5 提交热力图模块

## 14. 评分与规则设计

### 14.1 项目综合评分

用于项目排行和推荐。

建议公式：

- overall_score = activity_score * 0.45 + traffic_score * 0.30 + popularity_score * 0.15 + recency_score * 0.10

若仓库暂时无有效流量数据，则 `traffic_score` 不参与计算，其余权重按比例归一。

### 14.2 活跃度评分参考

- 近 30 天提交数
- 近 30 天活跃天数
- 最近提交时间
- 周趋势变化

### 14.3 流量评分参考

- 近 14 天 page views
- 近 14 天 unique visitors
- 近 14 天 clone 数

### 14.4 热度评分参考

- 仓库 star 数
- 仓库 fork 数

### 14.5 技术栈识别规则

- 若存在 react 和 vite，则识别为 React + Vite
- 若存在 next，则识别为 Next.js
- 若存在 vue 和 vite，则识别为 Vue + Vite
- 若存在 nestjs，则识别为 NestJS
- 若存在 prisma，则识别为 Prisma
- 若存在 tailwindcss，则识别为 Tailwind CSS
- 若存在 typescript，则识别为 TypeScript

## 15. 非功能需求

### 15.1 性能要求

- 首页首屏接口响应时间不超过 2 秒
- 单图表数据加载不超过 1 秒
- 数据同步时不阻塞前端读取

### 15.2 可维护性要求

- 前后端分层清晰
- 数据同步与数据展示解耦
- 技术栈识别规则可单独扩展
- 所有聚合逻辑可复用

### 15.3 可扩展性要求

- 可新增其他数据源
- 可扩展更多技术栈识别规则
- 可升级 SQLite 至 PostgreSQL
- 可扩展多用户账号

### 15.4 安全要求

- GitHub Token 不在前端暴露
- 敏感配置仅后端保存
- 私有仓库数据仅在本地或授权环境使用

### 15.5 当前访问控制实现

当前落地版本采用“单管理员 + 公开访客模式”：

- 公开访客可直接查看首页、项目列表、项目详情、技术栈分析、洞察中心和数据统计；
- 配置中心对访客保持只读；
- 管理员通过用户名密码登录后，才允许修改 GitHub 配置、保存同步策略、触发手动同步；
- GitHub Token 仅在服务端保存，前端仅返回 `hasToken` 等状态字段。

当前管理员账号来自环境变量：

- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`

当前站点支持通过环境变量 `GITHUB_TOKEN` 在服务启动时自动导入默认 GitHub 数据源配置。

### 15.6 当前配置中心行为

当前配置中心已落地的核心能力包括：

- 展示 GitHub 用户名、Token 连接状态、最近同步时间和当前同步状态；
- 管理员登录成功后可修改：
  - GitHub 用户名
  - GitHub Token
  - 邮箱别名
  - 是否同步私有仓库
  - 同步周期
  - 默认时间范围
- 访客模式下仅可查看状态，不可提交配置；
- 登录成功和失败时，前端会通过全局弹出提示反馈结果。

### 15.7 当前导航与项目详情行为

当前左侧导航中的“项目详情”入口不再是固定静态页，而是指向**当前已选择的项目详情**：

- 用户从首页项目排行进入详情页后，会记录当前项目；
- 用户从项目列表点击某个项目卡片进入详情页后，也会记录当前项目；
- 左侧“项目详情”菜单点击时，会优先跳回最近一次查看的项目详情；
- 若当前还没有已选项目，则前端弹出提示，引导用户先从首页或项目列表选择一个项目。

## 18. 风险与应对

### 18.1 数据源限制

- 风险：GitHub API 配额限制影响同步频率
- 应对：采用增量同步、缓存与定时策略

### 18.2 技术栈识别误差

- 风险：仅靠语言和文件推断可能存在误判
- 应对：加入置信度和来源字段，允许后期补充规则

### 18.3 流量数据不完整

- 风险：部分仓库缺少有效流量数据
- 应对：流量模块降级为可选展示，不影响主流程

### 18.4 数据规模增长

- 风险：提交数据积累后查询性能下降
- 应对：通过聚合表、索引和分页控制性能

## 20. 结论

本项目适合作为个人数据产品和作品集项目推进。其核心价值不在于单纯展示图表，而在于将分散的 GitHub 数据重组为一个可解释、可追踪、可比较的个人经营系统。第一版建议严格控制范围，优先完成 GitHub 单数据源、提交分析、技术栈画像和首页大屏，确保产品在短周期内形成完整闭环。
